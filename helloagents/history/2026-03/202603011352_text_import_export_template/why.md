# 变更提案: 文本管理下载/上传模板化闭环

## 需求背景
当前翻译人员需要离线处理文本，但系统缺少标准化下载/上传机制，导致线下编辑后回传存在字段错位、主键错配、状态混乱等风险。

## 变更目标
1. 提供“按筛选条件导出”的批量下载能力，满足离线翻译工作流。
2. 额外提供模板下载能力，避免格式不一致。
2. 提供严格校验的批量上传能力，防止错误文件污染线上数据。
3. 上传成功后统一更新译文、状态、编辑次数、修改时间，并保留更新记录可追溯。
4. 导出链路在 100 万级数据场景下保持稳定，避免内存溢出。

## 模板定义
- 导出列顺序（固定）: `编号`、`FID`、`TextId`、`Part`、`原文`、`译文`、`状态`
- 上传处理规则:
1. 先通过 `编号` 定位数据库文本记录。
2. 强校验 `编号/FID/TextId/Part` 与数据库完全一致，不一致立即报错。
3. 校验通过后，使用上传内容覆盖 `译文` 与 `状态`。
4. 每条记录执行 `editCount + 1`、更新时间更新，并写入 `text_changes`。

## 风险与约束
- 风险: 批量导入若部分成功会造成数据不一致。
- 风险: 大规模导出若采用一次性加载，可能触发 OOM。
- 约束: 必须“先全量校验后统一写入”，任一行失败则整体失败回滚。
- 约束: 导出必须采用流式读取和分批写入，禁止全量 `fetchall` 导出。
