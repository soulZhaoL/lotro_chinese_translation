# 变更提案: 旧版导入异常译文修复 SQL 生成

## 需求背景
此前使用旧版 `work_text/U46.1.xlsx` 导入数据库后，部分 `translatedText` 被截断，典型表现为协议文本中的 `{}` 或 `|` 丢失，导致译文只保留前半段，例如 `#1:{point[1]|points}` 被错误落成 `#1:{point[1]`。这不是展示层问题，而是导入时的协议分段/格式处理错误已经写入数据库。

当前用户已人工筛出疑似异常样本：
- `work_text/大括号差异样本_未修正.xlsx`
- `work_text/竖号差异样本_未修正.xlsx`

同时，修复依据也已明确：
- 必须按 `tools/valid_format/xlsx_to_insert_segmented.py` 的现行正则/协议规则解析新的 `work_text/U46.1.xlsx`
- 必须基于差异样本中的 `FID + TextId` 做精确匹配
- 必须最终生成面向 `text_main` 表的 `UPDATE SQL`

## 变更内容
1. 复用现有分段协议规则，从新的 `work_text/U46.1.xlsx` 重建正确的 `fid + textId + sourceText + translatedText + sourceTextHash + part` 数据集。
2. 读取两份差异样本，去重合并出待修复的目标键集合，并保留样本中的“当前错误译文”作为防误更新条件。
3. 基于 `fid + textId` 将新解析结果与差异样本精确匹配，生成只修复命中记录的 `UPDATE SQL`。
4. 输出未匹配清单、重复冲突清单与统计结果，禁止静默跳过。

## 影响范围
- **模块:** 文本导入工具、离线数据修复工具
- **文件:** `tools/valid_format/xlsx_to_insert_segmented.py`、新增修复脚本/配置、生成的修复 SQL 与核对结果
- **API:** 无
- **数据:** `text_main.translatedText`、必要时 `sourceText` / `sourceTextHash`

## 核心场景
### 需求: 按现行协议规则重建正确译文
**模块:** 离线修复工具
新 `U46.1.xlsx` 中的协议段必须按系统现行规则拆分，不能手写另一套正则。

#### 场景: 解析复合协议文本
输入单元格包含 `::::::`、`:::n:::`、`:::m-n:::` 三类协议头
- 解析结果必须保留完整字符串 `textId`
- 原文段与译文段数量必须严格一致
- 任意格式异常必须直接报错

### 需求: 仅修复差异样本命中的数据库记录
**模块:** SQL 生成
修复范围必须严格限定在差异样本内，不能对全量解析结果直接批量覆盖。

#### 场景: 合并两份差异样本
同一条记录同时出现在“大括号差异”和“竖号差异”样本中
- 最终待修复集合中只保留一条
- 必须保留样本原始错误译文供 SQL 防误更新使用

### 需求: 防止覆盖后来人工修正的数据
**模块:** SQL 生成
数据库中可能已有部分记录被人工修正，修复 SQL 不能无脑覆盖。

#### 场景: 数据库当前内容已不同于样本错误译文
同一 `fid + textId` 在样本中是错误值，但数据库内已被改正
- 生成的 `UPDATE SQL` 必须通过 `translatedText = 样本错误译文` 进行保护
- 未满足保护条件的记录应保留为“需人工复核”，不能强行更新

### 需求: 输出可审计的修复结果
**模块:** 运维交付
本次修复不是临时分析，而是要产出可执行、可复核的修复工件。

#### 场景: 交付修复脚本
修复流程执行完成后
- 必须生成正式 `UPDATE SQL`
- 必须生成匹配统计与未匹配明细
- 必须能说明每条 SQL 的来源是“样本错误记录 + 新版 Excel 正确记录”

## 风险评估
- **风险:** 只按 `fid + textId` 更新会覆盖数据库中后来人工修过的正确译文
  - **缓解:** 在 `WHERE` 中追加 `translatedText = 样本错误译文`，并输出“命中保护条件 / 未命中保护条件”统计
- **风险:** 新版 Excel 解析结果与差异样本存在一对多或零匹配
  - **缓解:** 命中关系必须唯一；遇到未匹配或重复匹配直接报错并输出清单
- **风险:** 两份样本存在重复且译文内容不一致
  - **缓解:** 合并样本时按 `fid + textId` 去重，并校验样本中的原文/错误译文是否一致，不一致则阻断
- **风险:** 修复脚本自行另写正则，和系统正式导入规则再次偏离
  - **缓解:** 直接复用或抽取 `xlsx_to_insert_segmented.py` 的核心解析逻辑，禁止复制后“魔改”
