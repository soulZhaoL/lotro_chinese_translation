# 变更提案: textId 字符串化与全链路修正

## 需求背景
当前系统把 `text_main.textId` 设计成了 `BIGINT`，并默认所有解析、查询、导入、版本迭代脚本都把 textId 当作纯数字处理。但实际源数据中存在协议头为复合标识的文本，例如 `126853056:::337429-296068:::[<--DO_NOT_TOUCH!--> <--DO_NOT_TOUCH!--> C]|||`，其真正的 textId 应为 `126853056:::337429-296068`，而不是仅取前缀数字 `126853056`。这意味着当前从导入、落库、筛选、详情查询，到版本迭代映射与继承，整条链路的主业务键都建立在错误前提上，已经不是单点 bug，而是数据建模错误。

## 变更内容
1. 数据模型修正：将 `text_main.textId` 由数值型改为字符串型，并同步修正相关索引、唯一约束、导入模板校验与文档定义。
2. 解析链路修正：统一 `tools/valid_format` 与 `tools/version_iteration_tool` 的 textId 提取规则，确保协议头中的完整 textId 被保留为字符串。
3. 接口与前端修正：前后端查询参数、类型定义、筛选输入、详情路由与导出上传链路全部按字符串 textId 处理。
4. 版本迭代修正：Step4/5/6/7 及映射表逻辑按字符串 textId 进行 join、分类、映射与迁移，避免版本比对错配。
5. 数据迁移与回灌策略：先修改字段类型，再基于原始源数据产出“错误 fid+textId → 正确 fid+textId”的对照结果，供后续执行全局 update 刷新；同时补充风险控制与执行边界。
6. 索引评估：结合约 80 万行数据规模，评估 `textId` 改为字符串后单列索引与组合唯一索引是否继续保留，以及是否需要调整索引策略。

## 影响范围
- **模块:** 数据模型、文本任务与翻译、文本校验、离线导入、版本迭代工具
- **文件:** `server/migrations/001_init.sql`、`server/routes/texts.py`、`server/routes/validate.py`、`web/src/modules/texts/**/*`、`web/src/pages/Translate.tsx`、`web/mock/*`、`tools/valid_format/*`、`tools/version_iteration_tool/*`
- **API:** `/texts`、`/texts/download`、`/texts/children`、`/texts/by-textid`、上传模板与相关查询链路
- **数据:** `text_main.textId` 字段类型、索引、唯一键、版本迭代 map 表结构、基于 `(fid,textId,part)` 的对比与映射逻辑

## 核心场景
### 需求: 协议段完整提取 textId
**模块:** 数据导入与版本迭代
对 `126853056:::337429-296068:::[xxx]` 这类协议段，系统必须将 `126853056:::337429-296068` 识别为完整 textId。

#### 场景: 解析复合协议头
输入文本段命中 `textId:::range:::[text]` 格式
- 解析结果中的 textId 必须保留完整协议头前缀与范围标识
- 不允许再把 textId 截断为纯数字前缀

### 需求: 按字符串 textId 精确查询与筛选
**模块:** 文本任务与翻译
列表筛选、子列表查询、详情编辑查询、导出与上传校验必须按字符串 textId 工作。

#### 场景: 通过复合 textId 查询
用户输入 `126853056:::337429-296068`
- 列表筛选能够命中对应记录
- 详情/编辑接口按 fid + textId 能精确返回唯一记录

### 需求: 版本迭代按真实业务键继承历史数据
**模块:** 版本迭代工具
Step5/6/7 必须基于真实 `(fid, textId, part)` 业务键执行比对、继承与映射，避免错继承、漏继承。

#### 场景: 版本升级后继承旧译文
新旧版本文本使用相同 fid、完整字符串 textId、part
- Step5 能正确识别 unchanged/changed/new
- Step6 能正确生成 oldId/newId 映射
- Step7 能将历史 `text_changes` 迁移到正确的新记录

### 需求: 生成错误/正确 textId 对照结果
**模块:** 数据模型与运维
针对已经入库且被截断的旧数据，需要从当前错误数据与原始源数据中产出可执行修复的对照结果。

#### 场景: 生成修复映射
准备执行 textId 字段类型改造后
- 系统必须输出“错误 fid + textId”与“正确 fid + textId”的对应关系
- 输出结果必须能直接用于后续全局范围 update 刷新
- 不允许人工猜测补值，匹配必须来自可验证的数据源与规则

### 需求: 字符串 textId 索引可支撑 80 万行查询
**模块:** 数据模型与性能
textId 改为字符串后，需要明确现有索引是否仍然适合当前查询模式与数据量。

#### 场景: 评估索引保留策略
表规模约 80 万行
- 需要评估单列 `textId` 索引是否继续保留
- 需要评估唯一索引 `(fid, textId)` 是否继续保留
- 需要评估 `(fid, part)`、`status`、`uptTime` 等现有索引是否足够支撑列表、详情、导出与版本迭代

## 风险评估
- **风险:** 旧库错误 textId 与源数据正确 textId 无法稳定一一匹配，导致批量 update 误伤
  - **缓解:** 映射生成必须基于 `fid`、协议解析结果及必要的辅助校验字段（如 sourceText/sourceTextHash/part）做精确匹配；生成后先抽样核对，再执行全局 update
- **风险:** 已落库旧数据只保留数值前缀，真实字符串 textId 已不可逆丢失
  - **缓解:** 先从原始 SQLite/XLSX 重算正确 textId，并与现库按可验证规则产出映射；若存在无法唯一匹配的数据，必须单独列出并阻断自动更新
- **风险:** `textId` 改成字符串后索引体积变大，可能拖慢写入或占用更多内存
  - **缓解:** 基于当前查询模式评估索引必要性；优先保留 `(fid, textId)` 唯一索引与 `textId` 单列索引，实施后用 explain/统计信息验证
- **风险:** 脚本、接口、前端对 textId 类型假设不一致，产生新一轮错配
  - **缓解:** 统一定义 textId 为字符串，逐层梳理类型、SQL、正则、接口参数与 mock 数据
- **风险:** `(fid,textId,part)` 相关索引和 map 表结构遗漏修改，导致版本迭代结果错误
  - **缓解:** 将 Step4/5/6/7 与 map 表纳入同一批改造与验证任务，按业务键做端到端校验
- **风险:** 现有查询接口中把 textId 作为 int 解析，导致复合值直接 422/400
  - **缓解:** 后端 query 参数、前端 QueryParams、筛选归一化逻辑统一改为字符串，并对空串做显式校验
