# 变更提案: 词典管理菜单对齐文本管理并补齐维护闭环

## 需求背景
当前词典管理模块仍采用轻量 `Form + Table + Modal` 实现，整体样式、搜索区组织方式、分页与操作体验均弱于文本管理模块，且缺少修改、导入、导出能力，无法支撑批量维护场景。

本次变更的目标不是只补一个按钮，而是把“词典管理”升级为与“文本管理”同等级的运营页面：界面风格统一、单条维护可编辑、支持模板导入导出，并补齐修改人和备注等运营信息。

## 产品分析

### 目标用户与场景
- **用户群体:** 词典维护人员、翻译运营人员
- **使用场景:** 日常增改词条、批量导入词典、导出给外部整理后再回传
- **核心痛点:** 当前只能新增与查询，不能批量处理，也看不到最近是谁改了什么

### 价值主张与成功指标
- **价值主张:** 让词典模块具备和文本管理一致的维护效率与可追溯性
- **成功指标:** 支持单条修改、模板导入导出、列表展示备注与修改人，页面交互风格与文本管理统一

### 人文关怀
导入错误需要给出明确行号和原因，避免维护人员在批量处理中反复试错。

## 变更内容
1. 词典管理页面 UI 全面对齐文本管理页面，统一为 ProTable 风格和搜索区操作栏。
2. 新增词条修改能力，允许修改译文 value、分类、备注，原文 key 只读。
3. 新增词典模板下载、导出、导入能力，导入按原文 key 执行覆盖或新增。
4. 列表新增备注、修改人字段展示，并保留更新时间。

## 影响范围
- **模块:** 词典管理、用户鉴权、前端 mock
- **文件:** `web/src/modules/dictionary/*`、`server/routes/dictionary.py`、`server/migrations/*`、`web/mock/*`、`tests/*`
- **API:** `/dictionary` 列表/新增/修改增强，新增模板下载、导出、导入接口
- **数据:** `dictionary_entries` 增加备注与修改人信息，并约束 `termKey` 唯一

## 核心场景

### 需求: 词典管理页面风格对齐
**模块:** 词典管理
词典管理菜单的搜索区、表格、按钮布局、分页与加载反馈对齐文本管理。

#### 场景: 进入词典管理页
用户进入词典管理页面
- 页面使用 ProTable 风格展示词典列表
- 搜索区左侧保留查询/重置，右侧放置导出、下载模板、导入、新增等操作按钮
- 列表具备统一分页、加载态和空态

### 需求: 单条修改词条
**模块:** 词典管理
用户可以在列表中直接进入修改弹窗，编辑译文、分类和备注。

#### 场景: 修改已有词条
用户点击“修改”
- 弹窗展示原文 key（只读）
- 用户可编辑译文 value、分类、备注
- 保存成功后刷新列表，并更新修改人和更新时间

### 需求: 模板导出与导入
**模块:** 词典管理
词典支持下载固定模板、导出当前筛选结果、按模板批量导入。

#### 场景: 下载模板
用户点击“下载模板”
- 系统下载仅含表头的 xlsx 文件
- 表头固定为：原文 key、译文 value、分类、备注

#### 场景: 导出当前筛选结果
用户设置筛选条件后点击“导出”
- 系统导出当前筛选结果
- 导出文件列顺序与模板完全一致

#### 场景: 导入词典文件
用户上传符合模板的 xlsx 文件
- 若原文 key 已存在，则覆盖该词条的译文、分类、备注
- 若原文 key 不存在，则新增词条
- 导入成功后返回新增数、覆盖数并刷新列表

### 需求: 列表补充运营信息
**模块:** 词典管理
列表增加备注与修改人显示，提升可追溯性。

#### 场景: 查看词典列表
用户浏览词典列表
- 可直接查看备注
- 可查看最后修改人
- 仍可查看更新时间

## 风险评估
- **风险:** 当前 `dictionary_entries.termKey` 无唯一约束，若历史上存在重复 key，会影响导入覆盖语义
- **缓解:** 迁移前先扫描重复 key；若存在重复，按最新更新时间保留一条并出具清洗策略，再补唯一索引
